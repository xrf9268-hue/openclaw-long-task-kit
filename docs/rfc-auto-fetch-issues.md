# RFC: Auto-Fetch Issues and Apply Fixes (v0.4)

**Status:** Draft
**Author:** LTK Contributors
**Created:** 2026-03-17
**Issue:** #23

## Summary

This RFC proposes adding a command (`ltk issue-loop`) that fetches open GitHub
issues, triages them by priority and size, and produces fix branches for human
review.  The system never merges automatically — every change requires explicit
human approval before it reaches the default branch.

## Motivation

Today the issue-to-fix cycle is fully manual:

1. Operator reads issue list.
2. Operator decides priority, creates a branch, implements, runs tests.
3. Operator opens a PR and waits for CI + review.

Steps 1-3 are repetitive and well-suited to automation.  A thin wrapper that
fetches issues and proposes fixes — without ever pushing to `main` — reduces
operator toil while preserving the review gate.

## Proposed Design

### Architecture

```
ltk issue-loop --repo owner/repo [--dry-run] [--max-issues N]
```

**Phases:**

1. **Fetch** — `GET /repos/{owner}/{repo}/issues` filtered by labels,
   milestone, and state.  Exclude issues that already have an open PR branch.
2. **Triage** — Sort by milestone priority (M0 > M1 > M2 > M3), then by size
   label (S > M > L).  Issues without size labels are skipped unless
   `--include-unsized` is passed.
3. **Plan** — For each selected issue, read the body and produce a structured
   implementation plan (file list, test plan, AC checklist).
4. **Implement** — Create a feature branch, apply TDD changes, run the local
   verification suite (`pytest`, `ruff`, `mypy`).
5. **Propose** — Open a draft PR with the plan, diff summary, and
   `Closes #<N>` in the body.  The PR stays in **draft** until a human marks
   it ready for review.

### Scope Boundary

LTK stays a thin wrapper.  `ltk issue-loop`:

- **Does** fetch issues, create branches, run local tests, open draft PRs.
- **Does NOT** merge PRs, approve reviews, push to `main`, or deploy.
- **Does NOT** modify files outside the repository working tree.
- **Does NOT** execute arbitrary shell commands from issue bodies.

### CLI Interface

```
ltk issue-loop \
  --repo owner/repo \
  --state <state-file> \
  --max-issues 1 \
  --dry-run
```

| Flag | Default | Description |
|------|---------|-------------|
| `--repo` | (required) | GitHub `owner/repo` |
| `--state` | (required) | Path to active task state file |
| `--max-issues` | `1` | Maximum issues to process per invocation |
| `--dry-run` | `false` | Fetch and triage only; no branches or PRs |
| `--include-unsized` | `false` | Process issues without a size label |
| `--exclude` | `[]` | Issue numbers to skip (repeatable) |
| `--milestone-priority` | `M0,M1,M2,M3` | Comma-separated priority order |

### State Integration

Each issue-loop run records its activity in the task state file:

```json
{
  "issue_loop": {
    "last_run": "2026-03-17T12:00:00Z",
    "issues_processed": [23],
    "issues_skipped": [3, 6],
    "prs_opened": [{"issue": 23, "pr": 56, "branch": "fix/issue-23"}]
  }
}
```

This prevents re-processing issues that already have open PRs and provides
an audit trail.

## Threat Model

### T1: Arbitrary Code Execution via Issue Body

**Threat:** A malicious issue body contains shell commands or code that the
automation blindly executes.

**Mitigation:**
- Issue bodies are treated as **read-only text** for planning purposes.
- The automation never runs `eval`, `exec`, `subprocess`, or any shell
  command derived from issue content.
- Implementation is generated by the agent's own reasoning, not by copying
  code from issue bodies.
- All generated code passes through the existing verification suite
  (`pytest`, `ruff`, `mypy --strict`) before any PR is opened.

### T2: Supply Chain Compromise via Dependency Changes

**Threat:** The automation adds or modifies dependencies in `pyproject.toml`.

**Mitigation:**
- The `dependency-audit` CI check runs on every PR.
- A policy constraint restricts the automation to stdlib-first changes.
- Any `pyproject.toml` modification in a generated PR triggers a mandatory
  human review flag.

### T3: Credential Leakage

**Threat:** Generated code or PR descriptions accidentally include tokens,
API keys, or other secrets.

**Mitigation:**
- The existing `sanitize` module scrubs sensitive patterns from all output.
- CI includes secret-scanning (GitHub's built-in push protection).
- `GITHUB_TOKEN` is passed via environment variable, never written to files.

### T4: Excessive API Usage / Rate Limiting

**Threat:** Unbounded loops exhaust GitHub API rate limits.

**Mitigation:**
- `--max-issues` defaults to 1 per invocation.
- The client checks `X-RateLimit-Remaining` headers and stops when below a
  threshold (default: 100 remaining).
- Exponential backoff on 403/429 responses (already in `GitHubClient`).

### T5: Unauthorized Repository Access

**Threat:** The automation operates on repositories the operator did not
intend.

**Mitigation:**
- `--repo` is a required explicit argument; no auto-discovery.
- The `GITHUB_TOKEN` scope limits accessible repositories.
- `--dry-run` mode allows previewing actions before execution.

### T6: Branch Pollution

**Threat:** Failed or abandoned automation runs leave stale branches.

**Mitigation:**
- Branch names follow a deterministic pattern: `ltk/issue-<N>`.
- A cleanup flag (`--prune-stale`) deletes branches whose PRs were closed
  without merging.
- Draft PRs are clearly labeled `[LTK auto-generated]`.

## User Consent and Review Workflow

### Consent Model

```
Operator opts in           Automation runs             Human reviews
─────────────────>  ─────────────────────────>  ─────────────────────>
  ltk issue-loop       fetch → plan → implement    review draft PR
  --repo X             branch → test → draft PR    mark ready / close
```

**Key principle:** The operator must take an explicit action (`ltk issue-loop`)
to start the process.  There is no background daemon, no webhook trigger, and
no automatic scheduling unless the operator configures a cron job.

### Review Gates

1. **Draft PR gate** — All PRs are created as drafts.  No reviewer is
   assigned automatically.  The operator must manually mark the PR as
   ready for review.

2. **CI gate** — Standard CI checks (lint, typecheck, test, audit) must
   pass before the PR can be merged.

3. **Human approval gate** — At least one human review approval is required.
   The automation never approves its own PRs.

4. **Merge gate** — Merging is always manual.  `ltk issue-loop` never calls
   `gh pr merge`.

### Audit Trail

Every automation run appends to the task state file and to the diagnostics
log (`ltk-diagnostics.jsonl`):

```json
{
  "event": "issue-loop",
  "timestamp": "2026-03-17T12:00:00Z",
  "repo": "owner/repo",
  "issues_fetched": 5,
  "issues_triaged": 3,
  "prs_opened": 1,
  "dry_run": false
}
```

### Rollback

If a generated PR introduces problems:

1. Close the PR (branch is auto-deleted if configured).
2. The original issue remains open for manual resolution.
3. Add the issue number to `--exclude` to prevent re-processing.

## Implementation Phases

### Phase 1: Fetch + Triage (v0.4.0)

- `ltk issue-loop --dry-run` fetches and sorts issues, prints the triage
  result.  No branches, no PRs.
- Tests: mock GitHub API responses, verify sorting logic.

### Phase 2: Branch + Implement (v0.4.1)

- Create feature branches and apply TDD-based fixes.
- Run local verification suite before proceeding.
- Tests: integration tests with temporary git repos.

### Phase 3: Draft PR (v0.4.2)

- Open draft PRs via GitHub API.
- Record activity in state file.
- Tests: mock PR creation, verify state recording.

## Open Questions

1. Should the automation support multi-repo operation, or stay single-repo?
2. What is the maximum acceptable issue complexity for automated fixes?
   (Proposed: Size S and M only; Size L requires human planning.)
3. Should there be a cost/token budget limit per issue?
